# Gita 开发规范（单元测试 & 代码注释）

> 本文档是《Gita 任务拆解》的执行前提，适用于 Phase 0 - Phase 6 的**所有**任务。AI Agent 在完成任何一个任务时，必须同时满足本文档的要求，而不仅是任务本身列出的"验收标准"。

---

## 1. 单元测试要求

### 1.1 覆盖范围（强制）

**每一个导出的函数/方法（public function/method）都必须有对应的单元测试**，无一例外。私有辅助函数如果包含独立的业务逻辑判断（如边界值计算），也应有测试，可通过测试导出函数间接覆盖。

每个函数的测试至少覆盖三类路径，缺一不可：

| 路径类型               | 说明                               | 示例                                              |
| ---------------------- | ---------------------------------- | ------------------------------------------------- |
| 正常路径（Happy Path） | 输入合法、符合预期的常规场景       | `GetStagedDiff()` 在有 staged 变更时返回正确 diff |
| 边界路径（Edge Case）  | 输入处于临界值、空值、极限值       | 分级函数在 999/1000/5000 等边界行数下的判断       |
| 异常路径（Error Case） | 输入非法、依赖不可用、外部调用失败 | 非 Git 仓库、API Key 缺失、LLM 超时               |

> **不允许只写正常路径的测试就视为任务完成**。如果某个函数确实不存在有意义的边界或异常场景，需在测试文件中用注释写明原因（例如"该函数为纯字符串拼接，无外部依赖，无异常分支"），而不是直接省略。

### 1.2 测试与业务代码同批提交

- 每个任务（如 T2.1、T1.5）在标记完成前，测试代码必须与业务代码在同一次提交中完成，不允许"先实现功能，测试留到后面补"。
- 《Gita 测试用例说明》文档中已列出的用例编号（如 `TC-T2.1-01`），测试函数命名建议直接体现编号，便于追溯，例如：
  ```go
  func TestClassifyDiff_TC_T2_1_01_ZeroLines(t *testing.T) { ... }
  ```

### 1.3 Mock 原则

- 涉及外部依赖（LLM API、文件系统、`$EDITOR`、网络请求）的函数，测试中必须使用 mock/stub，**不允许单元测试中发起真实网络请求**。
- 真实 LLM 调用仅在 T6.1 端到端集成测试中，且需要用环境变量控制是否跳过（如 `SKIP_REAL_LLM_TEST=1`），避免 CI 环境因缺少 API Key 而失败。

### 1.4 测试可读性要求

每个测试函数遵循 **Arrange-Act-Assert（准备-执行-断言）** 三段式结构，并用注释或空行分隔三部分，禁止把准备数据、调用逻辑、断言混在一起写成一坨：

```go
func TestClassifyDiff_Boundary5000(t *testing.T) {
    // Arrange: 构造恰好处于 Large 分级下边界的行数
    lineCount := 5000
    config := &Config{MaxDiffLines: 5000}

    // Act
    level := ClassifyDiff(lineCount, config)

    // Assert: 5000 行应判定为 Large（左闭区间，与需求文档 6.4 节一致）
    assert.Equal(t, Large, level)
}
```

### 1.5 覆盖率参考（非强制指标，但作为自查基准）

- `context` 模块（分级、摘要压缩逻辑）：核心判断分支要求 100% 覆盖，因为这是最容易出"差一行"bug 的地方
- `llm` 模块：正常/超时/空响应/格式错误四类路径均需覆盖
- `cmd` 交互层：四种按键路径（Y/e/r/n）+ 非法输入路径均需覆盖

## 2. 代码注释要求

### 2.1 注释的目的：解释"为什么"，而不是复述"做了什么"

**反例（禁止这样写）：**

```go
// 循环遍历文件列表
for _, file := range files {
    // 判断行数是否小于50
    if file.LineCount < 50 {
```

**正例：**

```go
// 单文件改动 < 50 行时，即便整体 diff 判定为 Large，
// 仍对该文件发送完整 diff 而非摘要（"抓大放小"原则，见需求文档 6.5 节），
// 避免小文件的关键改动因摘要压缩而丢失细节。
for _, file := range files {
    if file.LineCount < 50 {
```

### 2.2 必须写注释的场景（强制）

以下几类代码，**必须**有注释说明背景/原因，否则视为任务未完成：

1. **业务规则的边界值选择**
   例如分级阈值为什么是 1000/5000/15000，需注明"边界值参考需求文档 6.4 节，用户可通过 config.yaml 的 max_diff_lines 调整"。

2. **正则表达式**
   任何正则必须在上方用注释写清楚匹配的目标格式，并给一个示例输入：

   ```go
   // 匹配 git diff hunk header 中末尾的函数/类上下文文本
   // 示例输入: "@@ -10,7 +10,8 @@ func doSomething(ctx context.Context) error {"
   // 提取结果: "func doSomething(ctx context.Context) error {"
   var hunkHeaderPattern = regexp.MustCompile(`^@@ .*@@ ?(.*)$`)
   ```

3. **看似"不合常规"但有意为之的实现**
   例如 T4.1 中"用户连续输入 r 不限制次数"，需要在代码处注明原因，避免后来者误以为是遗漏而擅自加上限制：

   ```go
   // 注意：此处刻意不限制重新生成次数。
   // Gita 是开源工具，用户使用自己的 LLM Key，调用成本由用户自行承担，
   // 详见需求文档 5.3 节 Non-goals。
   ```

4. **降级/兜底逻辑**
   例如 hunk 上下文为空时的降级处理：

   ```go
   // Git 对部分小众语言（如 .proto）没有内置 funcname 识别规则，
   // 此时 hunk header 末尾无上下文文本，属预期情况而非解析失败，
   // 直接返回空字符串，上层会自动降级为"仅文件名+行数统计"展示。
   ```

5. **对外部命令/API 的调用**
   说明为什么用这个命令/参数而不是其他等价方案：

   ```go
   // 使用 --staged 而非直接 diff 工作区，
   // 因为 Gita 的设计原则是只处理用户已确认 add 的内容（见需求文档 6.3 节）。
   ```

6. **已知限制或 TODO**
   如果实现中存在已知但 MVP 阶段不解决的限制，必须显式注明，而不是留空：
   ```go
   // TODO(v1+): 当前摘要压缩仅做行数统计，
   // 未来可结合 tree-sitter 做更精确的语义级摘要，MVP 阶段不做（见需求文档 Non-goals）。
   ```

### 2.3 函数级注释（Docstring）

每个导出函数需要有函数级注释，格式：**一句话说明职责 + 关键参数说明（若不明显）+ 返回值的特殊情况说明**。


### 2.4 禁止事项

- 不写"显而易见"的注释（如 `// 返回 true` 写在 `return true` 上面）
- 不写过时后未同步更新的注释（宁可不写，也不要写错误的注释误导后来者）
- 不用注释替代清晰的命名（能通过改函数名/变量名说清楚的，不需要额外注释）

## 3. 任务完成的最终检查清单

AI Agent 在标记任意任务为"完成"前，除任务本身的验收标准外，额外自查以下几项：

- [ ] 该任务涉及的每个导出函数，是否都有对应单元测试（正常/边界/异常三类路径）
- [ ] 测试是否使用了 mock，未发起真实外部请求（LLM 调用类任务除外，且已加跳过开关）
- [ ] 边界值判断、正则表达式、降级逻辑、刻意的非常规实现，是否都写了"为什么"的注释
- [ ] 每个导出函数是否有函数级注释
- [ ] 是否存在"复述代码在做什么"的无意义注释，如有需删除或改写
