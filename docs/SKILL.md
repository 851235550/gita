# 目标

根据当前已暂存（staged）的代码变更生成高质量中文 Commit Message。

Commit Message 的目标不是复述 Diff，而是帮助未来的开发者理解：

- 为什么改
- 改了什么
- 带来了什么影响

生成的 Commit 应在 6 个月后仍然具有阅读价值。

---

# 执行步骤

## Step 1：读取变更

执行：

```bash
git status
git diff --staged
git diff --staged --stat
```

如果没有已暂存文件，返回：

```text
当前没有已暂存（staged）的代码变更，请先执行 git add。
```

停止后续流程。

**如果用户在触发时已经提供了背景说明**（例如"这是个 hotfix"、"为了兼容旧版 API"），记录下来，在 Step 4 生成时优先采纳，而不是仅从 diff 反推意图。

---

## Step 2：大 Diff 降级处理

先看 `git diff --staged --stat` 的汇总行数，再决定要不要读完整 diff：

| 变更规模 | 判断依据（`--stat` 汇总）     | 处理方式                                                                                                                                                                 |
| -------- | ----------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| 正常     | 改动行数（插入+删除）< 800 行 | 直接读取完整 `git diff --staged` 进行分析                                                                                                                                |
| 较大     | 800 ~ 3000 行                 | 优先读 `--stat` 掌握文件级改动分布，仅对改动集中的 1-2 个核心文件读完整 diff，其余文件依据 stat 中的增删行数 + 文件路径推断                                              |
| 过大     | > 3000 行                     | 不逐行读取 diff。仅基于 `--stat` 和 `--name-only` 生成粗粒度描述，并在输出末尾提示：`本次改动较大（约 N 行），建议使用 git add -p 分批暂存以获得更准确的 commit message` |

这一步的目的是避免把整份 diff 硬塞进上下文导致分析质量下降，而不是单纯为了省 token。

---

## Step 3：分析变更意图

重点关注：

- 本次修改试图解决什么问题
- 本次修改新增了什么能力
- 本次修改修复了什么缺陷
- 本次修改重构了什么结构
- 本次修改降低了什么风险
- 本次修改提升了什么能力

不要只关注代码细节，尽量推断修改背后的真实意图。

---

## Step 4：评估改动规模，选择输出结构

规模判断使用量化锚点，避免同一份 diff 每次判断结果不一致：

| 规模 | 判断标准（满足任一即算该档，取较高档）                      | 输出结构                  |
| ---- | ----------------------------------------------------------- | ------------------------- |
| 小型 | 涉及 1-2 个文件，且总改动 < 50 行                           | 仅标题                    |
| 中型 | 涉及 3-8 个文件，或单文件改动 50-300 行                     | 标题 + 变更项             |
| 大型 | 涉及 > 8 个文件，或新增了完整模块/目录，或单次改动 > 300 行 | 标题 + 背景 + 变更 + 影响 |

### 小型改动示例

```text
修复 MongoDB 副本集切换期间连接异常
```

### 中型改动示例

```text
重构 Kafka 消费者启动流程
- 统一消费者初始化逻辑
- 消除重复启动代码
- 优化异常处理流程
```

### 大型改动示例

```text
支持 MongoDB 副本集自动发现
背景：
当前配置依赖手工维护节点列表，
切主或扩容后容易失效。
变更：
- 增加副本集节点自动发现
- 支持 Primary 自动切换
- 优化连接初始化逻辑
影响：
- 降低运维成本
- 提升系统稳定性
```

---

## Step 5：与仓库现有风格对齐（可选但推荐）

执行 `git log --oneline -10` 快速查看最近的 commit 历史：

- 若明显遵循 Conventional Commit 格式（如 `feat(scope): ...`、`fix: ...`），在标题前加对应的 type 前缀，保持与历史一致
- 若历史 commit 是纯中文描述、无固定前缀格式，则沿用本 skill 默认的纯中文标题风格
- 无法判断（如仓库刚初始化、历史 commit 少于 3 条）时，默认使用纯中文标题，不强行加前缀

# Commit 编写原则

## 优先级

优先描述：

1. 为什么改
2. 改了什么
3. 带来什么影响

不要把重点放在：修改了哪些文件、调用了哪些函数、调整了哪些变量。

## 避免空洞表达

不要生成："更新代码" "优化逻辑" "修改功能" "调整实现" "修复问题" "代码重构"

必须明确指出：什么功能、什么问题、什么风险、什么收益。

## 避免文件级描述

不要生成："修改 mongo.go" "更新 kafka_consumer.go" "新增 test.go"

## 避免 Diff 翻译

不要逐行翻译代码，不要把 Diff 内容改写成自然语言，应该总结变更意图。

## 输出语言

使用简体中文（除非 Step 5 判断出仓库遵循英文 Conventional Commit 惯例，此时 type/scope 前缀部分保留英文，描述正文仍用中文）。

## 标题要求

简洁、准确、可独立阅读。推荐长度 10~30 个汉字。

---

# 生成前自检

输出前，问自己一个问题：

> **脱离当前代码上下文，另一位工程师 6 个月后看到这条 commit message，能否判断出改动的目的和影响？**

如果答案是"不能"，回到 Step 3 重新推断意图，而不是直接输出。

---

# 输出格式

```text
建议 Commit：
--------------------------------
<commit message>
--------------------------------
请回复：
1. 提交
2. 重新生成
3. 编辑
```

---

# 特殊要求

如果无法从 Diff 中推断出明确意图，优先生成：

```text
代码结构优化
- 简化实现逻辑
- 提升可维护性
```

不要编造不存在的业务背景，不要虚构需求，不要虚构收益。始终基于实际 Diff 进行推断。
